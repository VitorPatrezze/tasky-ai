## 🎯 Review Objectives

Every code review must verify:
1. **Correctness** — The code does what it's supposed to do
2. **Pattern Adherence** — The code follows established project patterns (see `.ai/tasky-project-patterns.md`)
3. **Code Quality** — The code meets quality standards (see `.ai/tasky-code-quality-guidelines.md`)
4. **Test Coverage** — All scenarios are tested (see `.ai/tasky-test-patterns.md`)
5. **Security** — No vulnerabilities are introduced
6. **Product Alignment** — The implementation meets product requirements (see `.ai/tasky-product-requirements.md`)

---

## ✅ Review Checklist

### 1. Correctness & Logic
- [ ] The implementation satisfies all stated requirements
- [ ] Edge cases are handled (null/undefined, empty collections, boundary values)
- [ ] Error paths are handled gracefully with appropriate error messages
- [ ] No infinite loops, race conditions, or deadlocks
- [ ] Data transformations produce correct output for all input types
- [ ] State mutations are intentional and predictable

### 2. Pattern Adherence
- [ ] New code follows the architectural patterns defined in `.ai/tasky-project-patterns.md`
- [ ] No new patterns introduced where existing ones apply
- [ ] API endpoints follow the established route and response conventions
- [ ] Data access follows the repository/service layer separation
- [ ] Dependency injection is used where established by the project
- [ ] File placement matches the project's directory conventions

### 3. Code Quality
- [ ] Naming conventions match `.ai/tasky-code-quality-guidelines.md`
- [ ] Functions/methods are focused and single-purpose
- [ ] No duplicated logic — shared concerns are extracted
- [ ] No dead code, commented-out blocks, or TODOs without tickets
- [ ] Types are explicit — no `any`, `object`, or untyped variables (where applicable)
- [ ] Comments explain "why", not "what" — code is self-documenting

### 4. Test Coverage Verification
> [!IMPORTANT]
> Ensuring complete test coverage is a critical part of every review. Missing tests are classified as **Warning** severity at minimum.

- [ ] Every new public method/function has corresponding unit tests
- [ ] Happy path scenarios are tested
- [ ] Error/failure scenarios are tested
- [ ] Edge cases are tested (empty inputs, max values, concurrent access)
- [ ] Integration points are tested (API endpoints, DB queries, external service calls)
- [ ] Test naming follows the conventions in `.ai/tasky-test-patterns.md`
- [ ] Mocks/stubs follow the established patterns in `.ai/tasky-test-patterns.md`
- [ ] No test logic leaks into production code

### 5. Security
- [ ] User input is validated and sanitized before processing
- [ ] Authentication and authorization checks are in place for protected resources
- [ ] Sensitive data (passwords, tokens, PII) is not logged or exposed
- [ ] SQL/NoSQL injection vectors are mitigated (parameterized queries)
- [ ] No hardcoded secrets, API keys, or credentials
- [ ] Dependencies are from trusted sources and up to date
- [ ] CORS, CSP, and other security headers are configured (if applicable)

### 6. Performance
- [ ] No unnecessary database queries (N+1 problem)
- [ ] Large data sets use pagination, streaming, or lazy loading
- [ ] Expensive computations are cached where appropriate
- [ ] No memory leaks (unclosed connections, unremoved listeners, growing collections)
- [ ] Async operations are properly awaited and error-handled

## 🚦 Severity Classification

All findings must be classified into one of these severity levels:

| Severity | Description | Action Required |
|----------|-------------|-----------------|
| **🔴 Critical** | Security vulnerabilities, data loss, crashes, severe logic errors, broken core functionality | Must be fixed before merge |
| **🟡 Warning** | Missing error handling, uncovered test scenarios, code smells, edge cases not handled, minor logic issues | Should be fixed before merge |
| **🔵 Info** | Naming improvements, formatting, stylistic suggestions, minor optimizations, documentation gaps | Can be deferred to a follow-up |
